## 📚 Git / GitHubの基本概念

### Gitとは
- **分散型バージョン管理システム**
- ソースコードの履歴管理、差分管理、並行開発、過去の状態への復元が可能

### GitHubとは
- Gitリポジトリをオンラインで管理・共有できるサービス
- プルリクエスト、コードレビュー、Issue管理、CI/CDなどチーム開発機能を搭載

### Git vs GitHub
| 観点 | Git | GitHub |
|-----|-----|--------|
| 性質 | バージョン管理システム | リモートリポジトリ管理サービス |
| 保存方式 | ローカル | オンライン（クラウド） |
| チーム開発 | 限定的 | スムーズ（PR、レビュー機能など） |

## 🚀 プロジェクトのはじめ方

### パターン1: ローカルコードをGitHub上で管理したい場合

```bash
# 1. ローカルリポジトリを初期化
git init

# 2. 変更をステージングしてコミット
git add .
git commit -m "初期コミット"

# 3. ブランチ名を変更（必要に応じて）
git branch -M main

# 4. リモートリポジトリを追加
git remote add origin <リモートリポジトリURL>

# 5. リモートリポジトリへプッシュ
git push -u origin main
```

**準備物**:
- GitHub上で新規リポジトリを作成
- 公開/非公開の設定
- 初期ファイル（README、.gitignore等）を必要に応じて作成

### パターン2: GitHub上のコードをダウンロードして編集したい場合

```bash
git clone <リポジトリのURL>
```

**クローンで自動実行される処理**:
- ローカルリポジトリの作成
- リモートリポジトリとの紐付け

## 👥 チーム開発の基本的な流れ（GitHub Flow）

### 1️⃣ リポジトリをクローンする
```bash
git clone <リポジトリURL>
cd <プロジェクト名>
```

### 2️⃣ ブランチを作成する
```bash
git checkout -b <ブランチ名>
```
**Point**: mainブランチに直接コミットせず、機能ごとにブランチを切る

#### ブランチの種類別役割

| ブランチ | 説明 | 命名例 |
|--------|------|--------|
| **main** | 本番環境。マージのみ。直接コミットなし | - |
| **develop** | 開発の中心。ここからさらにブランチを切る | develop/v1.0 |
| **feature** | 機能追加・変更・バグ修正用。developから派生 | feature/login-validation |
| **release** | リリース作業用。developから派生、完了後にmain/developにマージ | release/v1.0.0 |
| **hotfix** | 本番環境の緊急修正。mainから派生 | hotfix/v1.0.1/critical-bug |

### 3️⃣ 作業・コミットする
```bash
git add .
git commit -m "feat: ログイン機能のバリデーション追加"
```

#### コミットメッセージのベストプラクティス
- **何をしたか一目でわかる内容**
- **チームのルール（言語、prefix）に従う**
- **Conventional Commitsの形式を使う**: `<type>: <description>`

#### Conventional Commits - Prefix一覧

| Prefix | 説明 | 使用例 |
|--------|------|--------|
| **feat** | 新機能の追加 | feat: ユーザー登録フォームを実装 |
| **fix** | バグの修正 | fix: Nullポインター例外を修正 |
| **docs** | ドキュメント変更 | docs: README.mdを更新 |
| **style** | スタイル修正（意味に影響なし） | style: インデント修正 |
| **refactor** | リファクタリング（機能変更なし） | refactor: 変数名を統一 |
| **test** | テストコードの追加・修正 | test: ログイン処理のテストを追加 |
| **chore** | ビルド・依存関係などの管理 | chore: 依存ライブラリを更新 |
| **perf** | パフォーマンス改善 | perf: クエリを最適化 |
| **build** | ビルドシステムの変更 | build: Webpack設定を修正 |
| **ci** | CI/CDパイプラインの変更 | ci: GitHub Actionsワークフロー更新 |
| **revert** | 直前のコミットを取り消す | revert: 前のコミットを取り消し |

### 4️⃣ リモートにブランチをプッシュする
```bash
git push
```

### 5️⃣ プルリクエスト（PR）を作成する

#### PR作成時のポイント

**タイトル**:
- 何をしたPRなのか簡潔に記載
- 例: 「ログイン機能のバリデーション追加」

**説明欄 - テンプレート**:
```markdown
## 変更の概要
* 変更内容の概要
* 関連するIssueやプルリクエスト

## なぜこの変更をするのか
* 変更の理由
* 前提知識がなくても分かるように

## やったこと
* [x] 完了した項目
* [ ] 進行中の項目

## 変更内容
* UIの変更ならスクリーンショット
* APIの変更ならリクエスト・レスポンス

## 影響範囲
* ユーザーへの影響
* メンバーへの影響
* システムへの影響

## どうやるのか
* 使い方
* 再現手順

## 課題
* 悩んでいること
* レビューしてほしい箇所

## 備考
* その他の情報
```

**レビュワー設定**:
- チームのルールに従ってレビュー担当者を指定

### 6️⃣ コードレビューを受ける
- メンバーがPRを確認し、コメント・修正依頼
- 修正が必要 → 修正 → コミット → PRに自動反映
- OKが出たら → レビュワーが Approve（承認）

### 7️⃣ PRをマージする
- レビュー完了後、PRをマージ
- 通常はレビュワーやリポジトリ管理者が実行

### 8️⃣ ローカルを最新状態に保つ
```bash
git checkout main
git pull origin main
```

## ✅ 実務で役立つポイント

### mainブランチの保護ルール
```
✓ mainブランチに直接コミットしない
✓ PR経由でのみマージを許可
✓ レビュー必須化
✓ CI実行を必須化（GitHubのブランチ保護機能）
```

### 小さな単位でのコミット・PR

#### コミットのタイミング（細粒度が推奨）
- 新機能の一部（バリデーション追加、API呼び出し実装など）
- バグを1つ修正したとき
- ファイル構造や依存関係を更新したとき
- 実験的な作業を始める前のスナップショット

**メリット**:
- 変更の意図がわかりやすい
- コードレビューがしやすい
- バグ発生時の調査がしやすい

#### PRの頻度（1機能/修正ごと）
**目安**:
- 1つの小さい機能や修正ごとに1PR
- 例: 「ログインフォームのバリデーション追加」
- 例: 「プロフィール編集画面のレイアウト調整」
- 作業時間で半日〜1日以内が目安
- 2日以上かかりそうなら分割を検討

**メリット**:
- レビュー時間が短い
- マージコンフリクトが少ない
- CIが早く終わる
- リリース・ロールバックが安全

## 🔧 知っておくと便利なGitコマンド

### 🔍 git status - 状態確認の基本
```bash
git status
```
**習慣**: 作業中はこまめに実行し、現在の状態を把握

### 📝 git diff - 変更内容を確認
```bash
git diff              # ステージングしていない変更を表示
git diff --staged     # ステージングした変更を表示
```
**用途**: レビュー前に自分の変更を見直す

### 🔄 git log - コミット履歴を表示
```bash
git log --oneline --graph --decorate --all
```

**よく使うオプション**:
- `--oneline`: 1コミットを1行表示
- `--graph`: ブランチの分岐・マージをツリー表示
- `--decorate`: ブランチ名・タグも表示
- `--all`: すべてのブランチを対象

### 💡 git restore - 変更の取り消し
```bash
git restore <ファイル名>          # 作業ディレクトリの変更を取り消す
git restore --staged <ファイル名>  # ステージングを外す
```

### 🌟 git stash - 作業の一時退避
```bash
git stash              # 現在の変更を一時退避
git stash pop          # 一時退避した内容を戻す
```
**用途**: 作業途中で別の作業に切り替えたいとき

### 🚀 git fetch vs git pull
```bash
git fetch   # リモートの最新情報を取得（マージはしない）
git pull    # fetch + merge（ローカルにマージ）
```
**推奨方法**: `git fetch` → 差分確認 → `git merge`（安全）

## ❌ コンフリクト対応

### コンフリクト発生時の解決手順
```bash
# 1. コンフリクト状態の確認
git status

# 2. ファイルを手動で編集し、コンフリクト部分を解消
# （以下の記号を削除し、正しいコードに修正）

# 3. 修正後、ステージングしてコミット
git add <解決したファイル>
git commit

# 4. 必要に応じてプッシュ
git push
```

### コンフリクトマーカーの例
```
<<<<<<< HEAD
現在のブランチのコード
=======
マージ対象のブランチのコード
>>>>>>> feature/branch
```

## 📋 Issueの役割と使い方

### Issueとは
- チーム開発における「課題管理・タスク管理ツール」
- バグ報告、機能要望、作業メモ、ドキュメント改善など幅広く使用

### 実装の流れ

#### 1️⃣ Issueを作成する
```
タイトル: 内容がわかる簡潔なもの
説明欄: 背景、再現手順、スクショ、関連ファイル、期待する結果など
```
**💡 Tip**: Issueテンプレートを用意すると統一感が出ます（`.github/ISSUE_TEMPLATE/bug_report.md` など）

#### 2️⃣ メタデータを設定する
- **ラベル**: バグ、機能追加、リファクタリング、緊急、優先度高など
- **担当者（Assignee）**: そのIssueを対応する人
- **マイルストーン**: どのリリースで対応するか

#### 3️⃣ ブランチ作成時に関連付ける
```
ブランチ名の例:
- feature/123-ログイン機能
- fix/456-APIエラー対応
```

#### 4️⃣ PRにIssueをリンクさせる
```
PRの説明に以下のように記載すると、PR マージ時にIssueが自動でクローズされます:

close #123
fix #456
resolve #789
```

## 📌 まとめ・重要なポイント

1. **基本の流れ**: clone → ブランチ切る → コミット → プッシュ → PR作成 → レビュー → マージ
2. **mainブランチは常に安全**: ブランチ保護ルール、PR・レビュー・CI必須化
3. **小さな単位が最強**: 小さいコミット・PR = レビュー早い、バグ調査簡単、ロールバック安全
4. **コミットメッセージは重要**: Conventional Commitsフォーマットで統一感を出す
5. **Issueで課題管理**: チーム全体の進捗管理がスムーズ
6. **臨機応変に**: チームの方針に合わせながら、実務で学んでいくのが大切
